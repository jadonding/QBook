## 上下文

目标产品是一个跨平台记账 App，需要覆盖一木记账的核心能力，同时适配不同平台对系统能力的限制。移动端需要完整支持快速录入和智能能力，Web/Desktop 主要承担账单管理、统计分析和后台配置职责。

当前仓库尚未存在客户端代码，因此本次设计需要先确定整体架构、模块边界、数据模型和同步策略，避免后续实现阶段出现大规模返工。

约束条件：
- 使用单代码库支撑多平台交付。
- 核心记账、统计和资产能力必须支持离线使用。
- 自动记账、语音识别和截图导入等能力存在明显平台差异。
- 产品能力范围较大，需要按阶段上线。

## 目标 / 非目标

**目标：**
- 使用 Flutter 建立单代码库多平台客户端架构。
- 定义可支撑记账、资产、预算、共享和智能录入的数据模型。
- 采用本地优先和异步同步的方案，保证弱网和离线可用。
- 按能力域拆分模块，便于后续按阶段实现。
- 为 Android/iOS 完整体验和 Web/Desktop 降级体验提供统一规范。

**非目标：**
- 本次设计不包含具体 UI 视觉稿。
- 本次设计不定义服务端全部接口细节。
- 本次设计不承诺首版即实现自动记账、OCR 和行情同步的最终效果。
- 本次设计不覆盖营销、会员、支付订阅等商业化功能。

## 决策

### 决策 1：客户端采用 Flutter，移动端为主，Web/Desktop 为扩展平台

理由：
- 单代码库更适合当前从零开始的项目。
- 记账场景以表单、列表、统计图和本地数据为主，Flutter 适配度高。
- 可以先交付 Android/iOS，后续保留 Web/Desktop 管理能力。

备选方案：
- React Native：适合 React 团队，但 Web/Desktop 能力依赖更多生态组合。
- 原生多端：体验最佳，但开发和维护成本过高。

### 决策 2：采用 feature-first + 分层架构

理由：
- 业务模块较多，按能力域拆分比按技术层拆分更容易维护。
- 在 feature 内保留 presentation / application / domain / data 分层，可以控制依赖方向。
- 便于未来分别推进记账、预算、同步、智能录入等模块。

备选方案：
- 传统按 pages/services/models 全局分层：前期简单，但复杂度增长后容易耦合。

### 决策 3：本地数据库采用 Drift，本地优先写入

理由：
- 账单、统计和资产能力必须支持离线可用。
- Drift 适合结构化数据和复杂聚合查询。
- 本地优先写入后异步同步，能降低网络依赖并提升录入体验。

备选方案：
- 纯远端驱动：实现简单，但离线和弱网体验差。
- NoSQL 本地库：灵活，但聚合统计能力和约束不如关系型模型清晰。

### 决策 4：同步采用增量同步与软删除模型

理由：
- 账单、分类、账户等数据存在跨设备修改场景。
- 软删除有利于冲突处理、恢复和审计。
- 增量同步比全量覆盖更适合移动端和复杂数据集合。

备选方案：
- 全量覆盖同步：实现成本低，但冲突和流量成本更高。

### 决策 5：平台能力通过 adapter 层隔离

理由：
- 自动记账、语音、OCR、快捷入口高度依赖系统能力。
- 平台隔离可以减少业务模块直接依赖原生实现。
- 有利于 Web/Desktop 做能力降级。

备选方案：
- 直接在业务模块中调用平台插件：集成快，但后期难维护。

### 决策 6：按阶段上线，优先实现核心闭环

理由：
- 一木记账能力覆盖面大，不适合一次性实现。
- 先交付账本、账单、统计、预算和导出，可尽快建立最小可用版本。
- 智能能力和同步能力应在基础数据模型稳定后接入。

备选方案：
- 一次性覆盖所有能力：风险过高，验收边界不清晰。

## 风险 / 权衡

- [自动记账依赖系统权限和平台策略] → 先将其定义为移动端增强能力，首版不纳入最小闭环。
- [OCR 和语音识别准确率不足] → 提供人工修正和规则学习机制，避免把识别结果直接作为最终账单。
- [同步冲突导致数据不一致] → 对普通实体默认采用 last-write-wins，对金额敏感记录保留冲突记录供用户确认。
- [功能范围过大导致首版延期] → 按 capability 分阶段实现，严格控制 V1 范围。
- [Web/Desktop 与移动端能力差异过大] → 在规范中显式区分“全平台能力”和“移动端优先能力”。

## Migration Plan

1. 初始化 Flutter 工程和基础目录结构。
2. 建立本地数据库、核心实体和仓储接口。
3. 实现账本、账户、分类、账单和统计的基础闭环。
4. 增加预算、导入导出和本地备份。
5. 接入同步和共享。
6. 分阶段接入智能文字记账、语音、截图导入和自动记账。

回滚策略：
- 新功能以模块化方式增量引入。
- 智能能力和同步能力可以通过功能开关禁用。
- 数据库迁移必须保留版本化脚本和回退方案。

## Open Questions

- 服务端优先选择 Supabase 还是自建 API 还未最终确定。
- 智能文字记账是先走本地规则解析还是直接接入模型服务，需要结合成本再决策。
- 理财行情是否接第三方实时数据源，需要后续单独评估。
- 自动记账在 Android 的权限边界和 iOS 的快捷指令集成方式还需实现阶段验证。
